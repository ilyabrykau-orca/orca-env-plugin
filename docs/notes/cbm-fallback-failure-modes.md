# CBM Empty-Fallback Failure Modes — Research Artifact

## 1.1 Production Transcript Analysis (serena-dominates-transcript.json)

The first 10 tool calls in sequence: (1-5) ToolSearch for activate_project, search_code, find_symbol, get_symbols_overview, headroom_retrieve — all schema lookups. (6) `mcp__serena__get_symbols_overview` on containermonitor/ — the first *real* read, and it's Serena, not CBM. (7) `mcp__codebase-memory-mcp__search_code(pattern="containermonitor allocation", project="orca-runtime-sensor")`. (8) `mcp__codebase-memory-mcp__search_code(pattern="monitor.go memory pool", project="orca-runtime-sensor")`. (9-27) Nineteen consecutive `mcp__serena__find_symbol` calls reading individual symbols. The pivot point is between call 8 and call 9: after two CBM searches returned empty (likely because the project name `"orca-runtime-sensor"` doesn't match the indexed name `"Users-ilyabrykau-src-orca-runtime-sensor"`), Claude abandoned CBM entirely and switched to exhaustive Serena symbol reads. The 3rd CBM call at position 29 only came after all Serena reads completed — a "too late" retry that likely also used the wrong project name. Claude never attempted `list_projects`, `get_architecture`, or a corrected project name between the empty results and the Serena fallback.

## 1.2 CBM-Workflow Skill (skills/cbm-workflow/SKILL.md)

The skill does NOT teach what to do when a CBM call returns empty. It has no "empty result recovery" section, no mention of `list_projects` for project name discovery, and no warning that project names in the index are path-based (`Users-ilyabrykau-src-orca-runtime-sensor`) rather than short-names (`orca-runtime-sensor`). The skill does recommend starting with `get_architecture` for multi-symbol exploration, but presents it as an optimization ("one call replaces 4-6 round-trips"), not as a mandatory fallback when search_code fails. There is no anti-instruction discouraging Serena reads as a fallback for empty CBM results.

## 1.3 Serena-Workflow Skill (skills/serena-workflow/SKILL.md)

The skill does NOT forbid read-side use. It explicitly documents `find_symbol`, `get_symbols_overview`, `read_file`, and `list_dir` under a "Reading Code" section with examples. The framing is neutral — "Read symbol source by qualified name (preferred — token efficient)" with CBM listed first, then "Pre-edit read via Serena (when you need the symbol for editing next)" second. This makes Serena reads appear sanctioned for exploration, not just pre-edit contexts. There is no language like "only use these when you're about to edit" or "do not use Serena reads for code exploration."

## 1.4 SessionStart Injection (hooks/session-start-startup.sh)

The injected `<tool_routing>` block says: "Source read/search/navigate: mcp__codebase-memory-mcp__* only." This is imperative and clear. However, it says nothing about what to do when CBM returns empty — no retry guidance, no mention of project name discovery, no "if CBM fails, try different params before considering other tools." The rule is stated once; there's no escalation ladder. Claude can interpret "CBM returned empty" as "CBM can't help here" and legitimately seek another tool, because the routing block doesn't cover the empty-result case. The block also uses short project names ("orca-runtime-sensor") in the workspace detection, which Claude may copy into CBM calls without knowing the indexed name is the full path form.

## 1.5 Assert-Routing Library (tests/e2e/lib/assert-routing.sh)

The library uses `extract_tool_calls` from `verify-transcript.sh` which parses JSON transcripts extracting tool names from assistant turns. It operates on **tool names only** — it does NOT parse tool_result payloads, tool inputs, or user-turn content. Three centralized tool lists exist: `_CBM_READ_TOOLS`, `_SERENA_READ_TOOLS`, `_SERENA_EDIT_ADMIN_TOOLS`. The key assertion `assert_cbm_dominates_reads` counts CBM reads vs Serena reads and passes if CBM >= Serena. This catches the production case (3 CBM vs 19 Serena) but is a **ratio check**, not a **sequence check**. It cannot detect the specific failure mode of "empty CBM → immediate Serena fallback" if the overall ratio happens to favor CBM (e.g., 5 CBM reads, 3 Serena reads, but one CBM-empty → Serena-fallback pair hidden in the middle). The new assertion needs to parse both tool_use and tool_result entries, which requires extending the transcript format expectation to include user turns with result payloads.

## 1.6 CBM Project Names

Running `list_projects` confirms the canonical indexed project name for the runtime sensor is `"Users-ilyabrykau-src-orca-runtime-sensor"`, NOT `"orca-runtime-sensor"`. The production transcript used `project: "orca-runtime-sensor"` which would return empty results from CBM — this is a project-name mismatch, not a genuine "no results exist" case. The SessionStart hook detects the project as "orca-runtime-sensor" (short form) for Serena activation, and the skill examples also use short forms like `project="orca"`. This systemic mismatch between what the plugin tells Claude to use and what CBM actually indexes is a root cause contributor. All 11 indexed projects use the `Users-ilyabrykau-src-<name>` format.

## 1.7 Empty CBM Result Shapes

The production transcript fixture does not include tool_result entries (it only has assistant-side tool_use messages). However, based on CBM's response format from live calls, empty results take these forms: (a) `search_code` with no matches returns JSON with `"results": []` or a text message like `"0 results found for pattern '...' in project '...'"`. (b) For an invalid project name, CBM returns an error like `"Project 'orca-runtime-sensor' not found. Available projects: ..."`. (c) `search_graph` with no matches returns `"results": []` or `[]`. (d) `get_code_snippet` with an unknown qualified name returns `"Symbol not found: ..."`. For the new assertion, the key patterns to detect in tool_result content are: `"results":[]`, `"results": []`, `[]`, `"0 results"`, `"not found"`, `"No symbols found"`, `"No results"`, empty string, or project-not-found errors.

## 1.8 Test Runner Capability for tool_result Parsing

The current `launch-session.sh` uses `claude -p ... --output-format json --verbose`. The `--output-format json` flag produces a JSON array where each entry has `type: "assistant"` or `type: "user"`. Tool results appear in user-type entries as `{"type": "tool_result", "tool_use_id": "...", "content": [...]}`. The `extract_tool_calls` function in `verify-transcript.sh` only parses assistant entries for tool_use names. To implement `assert_cbm_retries_on_empty`, we need to also parse user entries for tool_result content, match them by `tool_use_id`, and inspect the result payloads. This is achievable with jq without infrastructure changes — the data is already in the transcript; we just need a new extraction function. No prerequisite gap exists.
